# HttpCookies


## Завдання 1: Дослідження HTTP Cookies через curl та DevTools

### Відповіді на питання:

#### 1. Що таке cookie і навіщо вони потрібні?
Cookies - це дані які сервер надсилає клієнту(зазвичай цим клієнтом виступає браузер) для того щоб в майбутніх запитах браузера до сервера він міг себе верефікувати за рахунок надсилання цих куків назад серверу разом з запитом.За рахунок цього сервер розуміє що це Аня а це Богдан

#### 2. Яка різниця між заголовками Set-Cookie (від сервера) та Cookie (від клієнта)?
Set-cookie це заголовок який відправляється сервером клієнту щоб клієнт зберіг ці дані.А Cookie це заголовок який клієнт кожний запит відправляє серверу щоб сервер знав хто він є.

#### 3. Чи можуть cookies працювати без браузера? Як curl їх обробляє?
Так, cookies можуть працювати без браузера, оскільки це звичайні текстові заголовки HTTP-протоколу. Будь-яка програма, що вміє працювати з мережею, може їх використовувати. Утиліта `curl` обробляє їх за допомогою спеціальних прапорців:
- Прапорець **`-c <file>`** (cookie-jar) змушує curl прочитати заголовок Set-Cookie від сервера та зберегти отримані куки у звичайний текстовий файл на диску.
- Прапорець **`-b <file>`** змушує curl зчитати куки з текстового файлу і автоматично додати їх у заголовок Cookie при відправці нового запиту.


## Завдання 2: Атрибути Cookies та їх призначення

### 1. Доповнена таблиця атрибутів cookies:

| Атрибут    | Призначення                                                         | Приклад значення            |
| ---------- | ------------------------------------------------------------------- | --------------------------- |
| Name=Value | Ім'я та значення cookie                                             | `sessionid=abc123`          |
| Domain     | Для яких доменів відправляти cookie                                 | `example.com`               |
| Path       | Для яких шляхів відправляти cookie                                  | `/api/v1`                   |
| Expires    | Коли закінчується дійсність cookie (конкретна дата і час)           | `Wed, 22 May 2027 10:00:00` |
| Max-Age    | Час життя cookie у секундах з моменту створення                     | `3600` (1 година)           |
| HttpOnly   | Забороняє доступ до cookie через JavaScript (захист від XSS-атак)   | Прапорець присутній (true)  |
| Secure     | Дозволяє відправку cookie тільки через захищене з'єднання HTTPS      | Прапорець присутній (true)  |
| SameSite   | Регулює відправку cookies при міжсайтових (Cross-Site) запитах       | `Strict` `Lax` `None,Secure`|

### 3. Відповіді на питання:

#### Чим відрізняються Session Cookies від Persistent Cookies?
- **Session Cookies (сесійні)** не мають встановлених атрибутів Expires або Max-Age. Вони зберігаються тимчасово в оперативній пам'яті браузера і автоматично видаляються, як тільки користувач повністю закриває браузер.
- **Persistent Cookies (постійні)** мають чітко визначений термін дії через Max-Age або Expires. Вони зберігаються на жорсткому диску комп'ютера і залишаються активними навіть після закриття браузера чи перезавантаження системи, поки не закінчиться їхній термін придатності.

#### Що означає HttpOnly і навіщо воно потрібно?
Атрибут **HttpOnly** — це прапорець безпеки, який вказує браузеру, що дані куки призначені виключно для передачі через мережу в HTTP/HTTPS заголовках. Він повністю приховує цю куку від клієнтських скриптів. Це потрібно для захисту від XSS-атак (Cross-Site Scripting), щоб зловмисники не могли викрасти сесії чи токени користувачів за допомогою шкідливого коду.

#### Чи може JavaScript з прямого доступу отримати HttpOnly cookie? Навіщо це потрібно?
**Ні, не може.** Якщо для куки встановлено атрибут HttpOnly, будь-яка спроба прочитати її через JavaScript (наприклад, за допомогою команди `document.cookie`) поверне порожній рядок або не покаже цю куку взагалі. Це критично важливо для безпеки: навіть якщо хакер знайде вразливість на сайті і зможе виконати свій скрипт, він фізично не зможе скопіювати сесійний токен користувача і вкрасти його аккаунт.

#### Яка різниця між Secure та HttpOnly?
Вони захищають дані на різних етапах:
- **Secure** захищає куку **під час транспортування (в мережі)**. Він гарантує, що браузер відправить куку лише через зашифрований протокол HTTPS, захищаючи її від перехоплення в мережі (наприклад, у відкритих Wi-Fi точках).
- **HttpOnly** захищає куку **всередині клієнта (в браузері)**. Він блокує доступ до куки з боку мови програмування JavaScript, захищаючи її від крадіжки через шкідливі скрипти на сторінці.
